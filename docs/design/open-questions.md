## ドメインルールの判断

### Q1. 完結済(completed)作品でのガード範囲
**回答: 作成のみ禁止し、既存 draft の publish / schedule は許す**(代案を採用)。
番外編・あとがきの後日公開といった運用を許容する。completed が禁止するのは「新規エピソードの作成」のみ。
→ `domain-model.md` 不変条件3、`state-machine.md` T1〜T3 のガードに反映。

### Q2. Work(作品)の削除を MVP に含めるか
**回答: 含める**(推奨どおり)。
カスケード削除(配下の Chapter / Episode / Cheer / Favorite / CheerComment を全削除)。
→ `domain-model.md` 不変条件8、`use-cases.md` U16 に反映。

### Q3. 配下にエピソードがある章の削除
**回答: 前の章に自動統合する**(代案Bを採用)。
配下エピソードは**直前の章の末尾**へ移動する(相対順序は保持)。**先頭の章を削除した場合は前の章が存在しないため、章なし区画(先頭)の末尾へ統合する**(この境界規則は反映時の補完。問題があれば指摘を)。
→ `domain-model.md` §2、`use-cases.md` U19 に反映。

### Q4. scheduled(予約中)エピソードの削除
**回答: scheduled からの直接削除も許す**(代案を採用)。
削除可能な状態は draft と scheduled。published のみ「まず取り下げ」をガイドする。
→ `state-machine.md` 遷移 T10 として追加。

### Q5. 予約公開の時刻精度
**回答: 分単位で可**(推奨どおり)。

### Q6. 応援(Cheer)の取り消しを許すか
**回答: 実物が取り消し可能なので、取り消しを許可する**(実物確認済み)。
Cheer は User × Episode で高々1つ、かつ取り消し(削除)・再応援が可能なトグル型。
→ `domain-model.md` 不変条件5、`use-cases.md` U6 に反映。

### Q7. 応援数・コメント数を ISR ページに焼き込むか
**回答: 目次・一覧の応援数はある程度古くてよいと割り切り、ISR の時間ベース定期 revalidate で遅れて反映する**(代案を採用)。
- 目次・一覧: 応援数を ISR に焼き込む。応援イベントでは on-demand 失効させず、時間ベースの revalidate で遅れて更新される。
- エピソードページのコメント欄・カウンタ: 従来どおり CSR 別フェッチ。
→ `state-machine.md` §5、`use-cases.md` §3 に反映。
