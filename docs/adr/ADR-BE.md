# ADR-BE: バックエンド設計の採用理由

**ステータス**: 採用
**最終更新**: 2026-07-06(概念設計・論理設計の完了に伴い起草)

## コンテキスト

本プロジェクトの設計上の主題の一つは、フレームワークの機能に依存しないバックエンド設計力(ドメインモデリング・状態遷移・不変条件・データ設計)の提示にある(knowledge doc §1)。本 ADR は、作品・エピソードの投稿・公開管理・閲覧機能(+応援・お気に入り)の概念設計・論理設計で下した判断のうち、**代替案との比較が意味を持つもの**を記録する。

個別仕様の正は `docs/design/` 配下の設計文書群(domain-model / state-machine / use-cases / db-schema / api-contract / isr-contract / validation-spec / implementation-rules)にあり、本 ADR は「なぜそう決めたか」だけを扱う。機能スコープの取捨選択(何を意図的に省略したか)は `docs/design/kakuyomu-scope-cut-list.md` が正である。

## 決定1: 予約公開は「独立した scheduled 状態 + 毎分のスケジューラ」で実現する

**決定**: Episode の公開状態を `draft / scheduled / published` の3状態の明示的な状態機械とし、予約公開は Laravel スケジューラ(毎分 tick)が `status = scheduled AND scheduled_at <= now` を走査して published へ遷移させる(案A)。公開時刻の精度は分単位とする。

**理由**:
1. 読み取りクエリが `status = 'published'` だけで閉じる。案B(下記)は全読み取り経路に時刻条件を漏れなく書く必要があり、**書き漏らし = リリース前コンテンツの漏洩**という高リスクの事故クラスを構造的に抱える。
2. ISR 前提では公開時刻到来を通知するジョブが**どのみち必要**になる(決定5)。ISR キャッシュは時刻の到来を自分では知らないため、ジョブなしでは公開が revalidate 間隔ぶん遅れる。
3. 「公開」がドメインイベントとして実在するため、published_at 設定・latest_published_at 更新(決定4)・ISR 再検証・(将来の)通知の発火点が1箇所に揃う。障害時も次 tick で自己修復する。

**却下した代替案**:
- **案B: published + 未来の published_at で表現し、公開判定をクエリに寄せる**: 状態を倒すジョブが正しさの前提にならず、公開時刻がクエリ評価時点で正確という利点がある。しかし理由2のとおり ISR 前提ではジョブが結局必要になり利点が消え、理由1の漏洩リスクだけが残る。

**結果**: スケジューラジョブ(`withoutOverlapping`、1件ずつ独立トランザクション)がインフラ要件に加わる — ただし Redis キュー/スケジューラ基盤は構築済みで追加リソースなし。公開時刻は指定から最大1分弱遅れることを仕様として受容する。

## 決定2: エピソードの順序は「密な連番 + 区画全体の書き換え」で表現する

**決定**: Chapter は Work 内で、Episode は所属区画(所属章、または章なし区画)内で 1..N の連番 `position` を持つ。並び替え・章間移動・章削除時の統合は、API が**目次の最終形全体**を受け取り(PUT /works/{uuid}/structure)、1トランザクションで該当区画の position を振り直す。「第N話」の話数番号はドメインに持たない(順序のみ。削除による歯抜けを許容)。

**理由**:
1. どんな並び替え操作(swap・任意位置挿入・章跨ぎ移動・章削除統合)も「最終形を書く」という**1パターンに落ちる**。中間状態の整合を考える必要がなく、自己修復的(次の全体書き換えで常に正規化される)。
2. カクヨムの並び替え UI は目次全体を組み替えて保存する形であり、「全体順序を送る」API と自然に一致する。
3. 負荷特性が全体書き換えを許す: 1作品のエピソード数は高々数百〜千のオーダーで、並び替えは Author の低頻度操作。

**却下した代替案**:
- **隙間つき整数(1000, 2000, ...)**: 単発移動が1行 UPDATE で済むが、隙間枯渇時のリバランスという第2の実装パスが必要。高頻度・大規模リスト向けの最適化で、本件では複雑さに見合わない。
- **分数ランク / LexoRank**: リバランス不要だがキーが人間に不可読になり、「目次の並び」というドメイン中心概念のデバッグ性を損なう。
- **連結リスト(prev_id)**: 読み取りに再帰が要り最悪。不採用。

**付随する決定 — position に UNIQUE 制約を張らない**: (1) MySQL のユニーク索引は NULL を重複とみなさず、章なし区画(chapter_id = NULL)では制約が効かない、(2) MySQL に遅延制約がなく、振り直し途中の値が制約に衝突する(負値退避などの迂回が必要になる)。一意性はトランザクション内の全体書き換え + Work 行の `lockForUpdate()` による直列化 + テストで担保する。

**結果**: 並び替え系のエンドポイントが1本(structure)に集約され、個別 move/reorder API 群が不要になる。話数番号の振り直しポリシーという問題自体が消滅する。

## 決定3: ID はハイブリッド方式(内部 BIGINT AUTO_INCREMENT + 公開 UUIDv7)とする

**決定**: 全テーブルの PK は BIGINT UNSIGNED AUTO_INCREMENT(内部専用)。URL / API に出るテーブル(users / works / chapters / episodes / cheer_comments)のみ、公開 ID として `uuid` カラム(CHAR(36)、UNIQUE、UUIDv7 をアプリ側生成)を別に持つ。内部 ID は API レスポンス・URL・ログに一切出さない(implementation-rules §5 で強制)。

**理由**:
1. InnoDB は全セカンダリ索引のリーフに PK を埋め込むため、UUID を PK にすると 36 バイトが索引・FK 全体に波及する。BIGINT なら 8 バイトで、特に cheers のような「小さい行 × 大量」のテーブルで差が大きい。
2. **DB を分割しない前提**(単一 RDS)では、UUID 単一 PK の唯一の構造的優位であるグローバル一意性が使われず、サイズ・速度の不利だけが残る。
3. 公開面は UUID なので、連番露出の弱点(推測可能性、総作品数・投稿ペースといった事業指標の漏洩)は同等に防げる。

**却下した代替案**:
- **UUIDv7 単一 PK(公開 ID 兼用)**: ID が1種類で済む単純さと、シャーディング・データ統合への耐性が利点。分散が視野に入るなら有力だが、本件の前提では理由2のとおり利点が空振りする。

**結果**: 「ID が2種類ある」規律コストを引き受ける: 外部からの参照は `WHERE uuid = ?`(各公開テーブルに UNIQUE 索引)、ルートモデルバインディングは uuid、API Resource 経由のシリアライズ必須。公開 uuid も v7 とする(PK でなくても UNIQUE 索引への挿入局所性に効く。生成時刻の漏洩は created_at を画面表示するため実害なし)。

## 決定4: 派生値は「一覧のソートキーだけ非正規化し、カウンタは非正規化しない」

**決定**: (1) 作品一覧のソートキー「最終公開日時」(= 配下 published Episode の published_at の最大値)は `works.latest_published_at` に**非正規化して持つ**。(2) 応援数・コメント数のカウンタキャッシュカラムは**持たず**、必要時(ISR 再生成時)に `COUNT(*)` する。

**理由**: 対照的な2判断だが基準は同じ — **読み書きの頻度差と、更新箇所が閉じているか**。
1. 最終公開日時: 読み(一覧 = サイトの顔・最頻アクセスのソート)は最多、書き(publish / unpublish)は低頻度。かつ更新イベントは状態遷移表とちょうど一致し、**どのみち ISR 再検証を発火する箇所**なので、更新漏れの監視点が同じ場所に集約される。都度 `MAX()` 集計はソートが索引に乗らず、公開作品数に比例して一覧が重くなる。
2. 応援数: 逆に書き(読者の応援)が高頻度で、非正規化すると1行カウンタの更新競合(ホットスポット)を自ら作ることになる。読みは ISR 再生成時(時間ベース revalidate、約5分間隔)だけなので、索引付き `COUNT` で十分。

**却下した代替案**: 最終公開日時の都度集計(理由1)/ 応援数のカウンタカラム(理由2)。応援が高頻度になった場合の counter cache 移行は将来の最適化として残す。

**結果**: latest_published_at の更新は publish / unpublish / 予約公開ジョブの同一トランザクション内で行う(implementation-rules §3)。応援数の表示は最大約5分遅れる(時間ベース revalidate の間隔)ことを UX 仕様として受容する。

## 決定5: ISR 再検証は「作品単位の一括タグ + キュー経由の発火」とし、公開処理の成否から分離する

**決定**: キャッシュタグは `works-list` / `work:{uuid}` / `episode:{uuid}` の3種のみとし、**エピソードページにも `work:{uuid}` を併記**して作品単位で一括失効させる(prev/next の波及先である「読書順で隣のページ」を精密に特定しない)。発火は DB コミット後(afterCommit)に Redis キューへ投入したジョブが Next.js の再検証エンドポイント(East-West / 共有シークレット)を叩く。リトライ3回、最終失敗しても公開処理は成功のまま。

**理由**:
1. ISR の失効は**遅延再生成**であり、過剰失効のコストは「失効後に読まれたページの再生成1回」に過ぎない。一方、隣接特定ロジック(削除・非公開を挟んだ読書順の隣、章跨ぎ、統合)のバグは**古い prev/next が残り続ける**発見しにくい事故になる。正しさを単純さで買う。
2. 同期 HTTP での発火は公開 API のレスポンスタイムに Next.js の応答を混ぜ、失敗時に「公開は成功したのか」を曖昧にする。状態遷移の成否と再検証の成否を分離し、取りこぼしの安全網は時間ベース revalidate に持たせる。
3. コミット前に失効させると、再生成が旧データを読んで古いページを焼き直す競合が起きるため afterCommit とする。

**却下した代替案**:
- **イベントごとの精密失効(隣接ページの列挙)**: 節約効果が再生成1〜2回分しかなく、理由1のリスクに見合わない。
- **同期呼び出し / 失敗時ロールバック**: 「Next.js が落ちていると小説を公開できない」という不合理な結合を生む。

**結果**: Laravel 側の発火は Event → Listener → ジョブの一経路に一元化される(implementation-rules §4)。この設計の前提として、HTML キャッシュの階層は Next.js(ElastiCache)の1層のみとし、CloudFront には HTML を置かない(isr-contract §5。基盤側の判断は ADR-INFRA 決定8 と整合)。

## 決定6: 状態遷移をアクション型 POST(URL に動詞)で表現する

**決定**: ガード付きの状態遷移は `POST /episodes/{uuid}/publish` のようなアクション型エンドポイントで表現する。ガードのない2値スイッチ(Work の visibility)のみ `PUT /works/{uuid}/visibility` の値置換とする。

**理由**:
1. 遷移はガードと副作用(published_at 設定、派生値更新、ISR 再検証)を持つ**ドメインイベント**であり、`PATCH {status: "published"}` の形にすると「status という値の更新」に見えてガードの所在が曖昧になる。バリデーション違反(422)とドメインガード違反(409)の区別も API 面から消える。
2. ユビキタス言語の動詞(publish / schedule / unschedule / unpublish / complete / reopen)が URL・UseCase クラス・ドメインメソッド・テスト名まで一貫し、状態遷移表(T1〜T10)と API が1:1で突き合う。

**却下した代替案**: 全部 PATCH によるリソース状態更新(REST 純粋主義)。理由1・2のとおり、本件では「リソースの表現の転送」より「遷移の実行」が API の実態であるため不採用。

**結果**: エンドポイント数は増えるが、各エンドポイントの責務(=1遷移)が最小になり、UseCase 1クラス=API 1本=遷移表1行の対応が成立する。

## 決定7: タイムゾーンを全層 Asia/Tokyo に統一する

**決定**: Laravel config・php.ini・MySQL の DATETIME 保存値・予約公開 UI・API の日時表現(ISO 8601 +09:00)を、すべて Asia/Tokyo に統一する。UTC 保存 + 表示時変換の層は持たない。

**理由**: 日本語専用サービスで多タイムゾーン要件がなく、予約公開の比較(`scheduled_at <= now`)・スケジューラ・表示が単一 TZ で閉じる。TZ 変換層は「どの層で変換済みか」という追跡コストを常に生み、単一 TZ 前提ではリスクだけの投資になる。

**却下した代替案**: **UTC 保存 + 表示変換(一般的定石)**。多リージョン・多 TZ のサービスでは正しい判断であり、実サービス化・国際化の際は最初に見直すべき決定であることを明記して残す。

**結果**: 全層の TZ 設定が一致していることが前提になるため、config の TZ をテストで固定・検証する。DATETIME(TIMESTAMP でなく)を使い、DB 層の暗黙 TZ 変換を避ける。

## 全体の結果(この ADR の帰結)

- 公開ライフサイクルは「明示的な状態機械 + ジョブで倒す」(決定1)、順序は「全体書き換え」(決定2)という、**クエリ側を単純に保ち、書き込み側に複雑さを寄せる**方向で統一されている。読まれる回数が書かれる回数より圧倒的に多い閲覧サイトの特性に沿った配分である。
- 決定4・5 も同じ思想の適用: 読み(一覧ソート)のための非正規化は引き受け、書き(応援)のホットスポットは作らず、失効は粗くして遅延再生成に任せる。
- ID(決定3)・TZ(決定7)は「単一 DB・単一地域」という前提を明示的に採った判断であり、前提が変わったときに見直す箇所として本 ADR に固定されている。
